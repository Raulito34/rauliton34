## 1. 바이브코딩이란?

**바이브코딩(Vibe Coding)** 이란 코드를 직접 한 줄씩 작성하는 대신, AI에게 자연어(한국어/영어)로 "이런 걸 만들어줘"라고 요청하면서 소프트웨어를 만드는 방식입니다.

기존 개발 방식과의 차이를 비유하자면:

| | 기존 개발 | 바이브코딩 |
|---|---|---|
| **비유** | 직접 요리하기 | 셰프에게 "이런 맛으로 해주세요"라고 주문하기 |
| **필요한 것** | 프로그래밍 언어 숙달 | 원하는 결과물에 대한 명확한 설명 |
| **작업 방식** | 코드를 한 줄씩 작성 | AI와 대화하며 방향 제시 |

핵심은 **"무엇을 만들고 싶은지"를 잘 설명하는 것**입니다. 코드 문법을 몰라도 됩니다.

## 2. 어떤 웹사이트를 만들었나?

**선아트센터(SUN ART CENTER)** — 서울 인사동에 위치한 아트센터의 공식 웹사이트입니다.

주요 기능:
- **전시 안내**: 현재/예정/지난 전시 정보 열람
- **공간 안내**: B1F~4F 전시관별 상세 정보
- **대관 시스템**: 일정 확인 → 신청서 작성 → 관리자 승인
- **소식/공지**: 공지사항과 뉴스 관리
- **문의하기**: 방문자 문의 접수
- **관리자 기능**: 전시/뉴스 등록·수정·삭제, 대관 신청 승인

## Step 1: 프로젝트 뼈대 만들기

> **AI에게 한 요청**: "아트센터 웹사이트를 만들고 싶어. React랑 Express로 해줘"

### 무슨 일이 일어났나?

AI가 프로젝트의 기본 폴더 구조를 자동으로 생성했습니다.

```
rauliton34/              ← 프로젝트 전체 폴더
├── client/              ← 프론트엔드 (사용자가 보는 화면)
│   ├── src/             ← 소스 코드
│   ├── package.json     ← 사용하는 라이브러리 목록
│   └── vite.config.ts   ← 빌드 설정
├── server/              ← 백엔드 (데이터 처리 서버)
│   ├── src/             ← 서버 코드
│   ├── prisma/          ← 데이터베이스 설정
│   └── package.json     ← 서버 라이브러리 목록
└── package.json         ← 전체 프로젝트 설정
```

### 쉽게 설명하면?

웹사이트는 크게 두 부분으로 나뉩니다:

- **프론트엔드 (client)**: 사용자가 브라우저에서 보는 화면. 마치 **식당의 홀(매장)** 과 같습니다.
- **백엔드 (server)**: 데이터를 저장하고 처리하는 서버. 마치 **식당의 주방** 과 같습니다.

손님(사용자)은 홀(프론트엔드)에서 메뉴를 보고 주문하지만, 실제 요리(데이터 처리)는 주방(백엔드)에서 이루어집니다.

## Step 2: 백엔드 — 데이터를 저장하는 서버 만들기

> **AI에게 한 요청**: "전시, 공간, 대관, 뉴스, 문의 데이터를 관리하는 API를 만들어줘"

### 2-1. 데이터베이스 설계 (Prisma + PostgreSQL)

먼저 "어떤 데이터를 저장할지"를 정의했습니다. 이것을 **스키마(Schema)** 라고 합니다.

엑셀 시트를 만드는 것과 비슷합니다. 각 시트(테이블)에 어떤 열(항목)을 넣을지 결정하는 겁니다:

| 테이블 이름 | 저장하는 데이터 | 주요 항목 |
|---|---|---|
| **Exhibition** | 전시 정보 | 제목, 작가, 설명, 기간, 층, 이미지, 상태 |
| **Space** | 전시 공간 정보 | 이름, 층, 면적, 천장높이, 수용인원, 특징 |
| **Rental** | 대관 신청 | 신청자, 공간, 기간, 목적, 상태(대기/승인/거절) |
| **RentalPricing** | 대관 요금표 | 공간별 1~4주 요금 |
| **News** | 소식/공지 | 제목, 내용, 분류(공지/뉴스) |
| **Contact** | 문의 접수 | 이름, 이메일, 제목, 내용 |

이 설계를 `schema.prisma`라는 파일 하나에 작성하면, **Prisma**라는 도구가 자동으로 데이터베이스 테이블을 만들어줍니다.

### 2-2. API 만들기 (Express.js)

API는 프론트엔드와 백엔드가 소통하는 **약속된 주소**입니다.

예를 들어:
- `GET /api/exhibitions` → "전시 목록 줘" → 서버가 전시 데이터를 보내줌
- `POST /api/rentals` → "대관 신청할게" → 서버가 신청 데이터를 저장함
- `DELETE /api/news/3` → "3번 뉴스 삭제해" → 서버가 해당 뉴스 삭제

총 **23개 이상의 API**를 만들었고, 관리자 전용 API에는 비밀번호 확인 절차를 추가했습니다.

### 2-3. 초기 데이터 넣기 (Seed)

빈 데이터베이스에 샘플 데이터를 넣는 작업입니다:
- 5개 전시 공간 (B1F~4F)
- 공간별 대관 요금표
- 4개 전시 샘플
- 5개 뉴스/공지 샘플

이렇게 하면 개발 중에 빈 화면이 아닌, 실제와 비슷한 화면을 보면서 작업할 수 있습니다.

## Step 3: 프론트엔드 — 사용자가 보는 화면 만들기

> **AI에게 한 요청**: "센터소개, 전시안내, 공간안내, 대관안내, 소식, 문의 페이지를 만들어줘. Tailwind CSS로 스타일링해줘"

### 3-1. 사용한 기술

| 기술 | 역할 | 비유 |
|---|---|---|
| **React** | 화면을 만드는 도구 | 레고 블록 (조립식 UI) |
| **TypeScript** | 코드 오류를 미리 잡아주는 도구 | 맞춤법 검사기 |
| **Tailwind CSS** | 디자인을 입히는 도구 | 옷장에서 옷 고르기 |
| **React Router** | 페이지 이동을 관리 | 건물의 엘리베이터 |
| **Vite** | 개발 서버와 빌드 도구 | 공사장의 크레인 |

### 3-2. 만든 페이지들 (총 17개)

```
홈페이지 (/)
├── 센터소개 (/about)
│   ├── 인사말
│   ├── 건축/공간
│   └── 오시는 길
├── 전시안내 (/exhibition)
│   ├── 현재전시
│   ├── 예정전시
│   └── 지난전시
├── 공간안내 (/spaces)
│   ├── B1F 제1전시관
│   ├── 1F 제2전시관
│   ├── 2F 제3전시관
│   ├── 3F 제4전시관
│   └── 4F 다목적홀
├── 대관안내 (/rental)
│   ├── 대관절차
│   ├── 대관료
│   ├── 대관규약
│   ├── 대관신청 (캘린더+신청서)
│   └── 신청내역 (관리자)
├── 소식 (/news)
└── 문의 (/contact)
```

### 3-3. React의 핵심 개념: 컴포넌트

React에서는 화면을 **컴포넌트(Component)** 라는 조각으로 나눠서 만듭니다.

예를 들어 이 웹사이트의 모든 페이지는:

```
Layout (전체 틀)
├── Header (상단 네비게이션)
├── 페이지 내용 (각 페이지마다 다름)
└── Footer (하단 정보)
```

이렇게 만들면 Header와 Footer를 한 번만 만들어두고, 가운데 내용만 바꾸면 됩니다. 마치 액자 틀은 고정해두고 그림만 교체하는 것과 같습니다.

## Step 4: 프론트엔드와 백엔드 연결하기

> **AI에게 한 요청**: "프론트엔드에서 백엔드 API를 호출하는 서비스 레이어를 만들어줘"

### 어떻게 연결되나?

```
[사용자 브라우저]  →  [프론트엔드 (3000번 포트)]  →  [백엔드 (4000번 포트)]  →  [데이터베이스]
     ↑                                                                              ↓
     └──────────────────────── 화면에 데이터 표시 ←──────────────────────────────────┘
```

프론트엔드의 `api.ts` 파일이 이 연결 역할을 합니다. 예를 들어:

```
사용자가 "전시안내" 페이지를 열면
→ 프론트엔드가 백엔드에 "GET /api/exhibitions" 요청
→ 백엔드가 데이터베이스에서 전시 목록을 꺼내옴
→ 프론트엔드가 받은 데이터로 카드형 목록을 화면에 그림
```

### 개발 중의 연결 (Proxy)

개발할 때는 프론트엔드(3000번)와 백엔드(4000번)가 서로 다른 포트에서 돌아가는데, **Vite의 프록시** 설정을 통해 자동으로 연결됩니다. 택배 중계소처럼 `/api`로 시작하는 요청을 백엔드로 전달해주는 역할입니다.

## Step 5: 배포 — 세상에 공개하기

> **AI에게 한 요청**: "Vercel로 프론트엔드, Railway로 백엔드 배포해줘"

### 왜 두 곳에 나눠서 배포하나?

| 서비스 | 배포 대상 | 이유 |
|---|---|---|
| **Vercel** | 프론트엔드 | 정적 파일(HTML/CSS/JS) 호스팅에 특화, 무료 |
| **Railway** | 백엔드 + DB | Node.js 서버와 PostgreSQL DB 운영에 적합 |

비유하면, 식당의 **인테리어(프론트)** 는 전문 인테리어 업체(Vercel)에, **주방 설비(백엔드)** 는 주방 전문 업체(Railway)에 맡긴 것입니다.

### 배포 과정

1. **vercel.json** 작성: "client 폴더를 빌드해서 배포해줘"
2. **Dockerfile** 작성: "server 폴더를 Docker 컨테이너로 만들어줘"
3. **railway.json** 작성: "Docker로 빌드해서 배포해줘"

GitHub에 코드를 올리면, Vercel과 Railway가 자동으로 최신 코드를 가져가서 배포합니다.

## Step 6: 배포 후 문제 해결

배포 후에는 예상치 못한 문제들이 발생했고, 이를 하나씩 해결했습니다. **이 과정이 실제 개발에서 가장 많은 시간이 소요되는 부분입니다.**

### 문제 1: Railway 빌드 실패 (4번의 시도)

| 시도 | 문제 | 해결 |
|---|---|---|
| 1차 | Nixpacks 빌드 도구가 서버를 인식 못함 | Nixpacks 설정 파일 추가 |
| 2차 | 여전히 빌드 실패 | Dockerfile로 전환 |
| 3차 | Dockerfile 위치 문제 | 서버 폴더에 Dockerfile 배치 |
| 4차 | Prisma 스키마 인식 실패 | Dockerfile에서 복사 순서 조정 |

### 문제 2: CORS 에러 (2번의 수정)

**CORS란?** 브라우저의 보안 정책으로, "다른 주소의 서버에 요청을 보내도 되는지" 확인하는 것입니다.

- Vercel(프론트)의 주소: `https://rauliton34.vercel.app`
- Railway(백엔드)의 주소: `https://xxx.railway.app`

서로 다른 주소이므로 브라우저가 차단합니다. 해결법: 백엔드에서 "이 주소에서 오는 요청은 허용해"라고 설정.

추가로 주소 끝에 `/`가 붙는 미세한 차이 때문에 한 번 더 수정이 필요했습니다.

### 문제 3: Vercel 빌드에서 Python 감지

프로젝트에 별도의 Python 파일이 있어서 Vercel이 "이건 Python 프로젝트야"라고 오인. Node.js로 설치하라고 명시적으로 지정해서 해결.

### 문제 4: TypeScript 타입 에러

배포 전 빌드 과정에서 TypeScript가 타입 불일치를 발견. 예를 들어 전시 상태를 `'upcoming'`만 기대하는 변수에 `'current' | 'upcoming' | 'past'` 값이 들어가는 문제. 타입 정의를 올바르게 수정하여 해결.

> **교훈**: 배포는 "한 번에 성공"하는 경우가 거의 없습니다. 에러 메시지를 읽고 → 원인을 파악하고 → 수정하는 반복이 핵심입니다.

## Step 7: 기능 확장 — 대관 시스템과 관리자 기능

> **AI에게 한 요청**: "대관 현황을 캘린더로 보여주고, 관리자가 전시와 뉴스를 관리할 수 있게 해줘"

### 7-1. 대관 캘린더 (StatusPage)

8주 단위로 각 전시관의 대관 가능 여부를 보여주는 표:

```
기간              | B1F     | 1F      | 2F      | 3F      | 4F
2026.02.10~02.16 | ☑ 선택  | 심사중  | ☑ 선택  | 대관완료 | ☑ 선택
2026.02.17~02.23 | ☑ 선택  | 심사중  | ☑ 선택  | ☑ 선택  | 대관완료
```

사용자가 원하는 날짜/공간을 체크하면 → 신청 페이지로 이동합니다.

### 7-2. 관리자 기능

관리자 비밀번호(x-admin-code)를 입력하면:
- **전시**: 등록 / 수정 / 삭제 (이미지 업로드 포함)
- **뉴스**: 공지사항, 센터 소식 등록 / 수정 / 삭제
- **대관**: 신청 목록 확인, 승인/반려 처리

처음에는 대관 데이터를 브라우저에만 저장(localStorage)했다가, 나중에 서버 DB로 이전하여 여러 사용자가 공유할 수 있게 개선했습니다.

## 바이브코딩에서 배운 점

### 잘 된 점

1. **속도**: 전통적으로 수 주가 걸릴 풀스택 웹사이트를 매우 빠르게 완성
2. **품질**: TypeScript 타입 안전성, 반응형 디자인, 관리자 인증 등 전문적 수준
3. **반복 개선**: "이것 좀 바꿔줘" 한마디로 여러 파일을 동시에 수정 가능

### 주의할 점

1. **배포 문제는 여전히 어려움**: 인프라 설정(Docker, CORS, 환경변수)은 여러 차례 시행착오 필요
2. **명확한 요구사항이 중요**: "모던하게 해줘"보다 "Pace 갤러리 참조해서, 블랙/화이트 톤으로"가 훨씬 좋은 결과
3. **검토 필수**: AI가 만든 코드도 빌드 에러나 타입 에러가 발생할 수 있음 → 반드시 빌드 테스트
4. **점진적 접근**: 한 번에 모든 것을 만들지 말고, 핵심 기능부터 → 확장 → 디자인 순으로 진행

### 바이브코딩에서 가장 중요한 것

> **"무엇을 원하는지"를 명확하게 설명하는 능력입니다.**
>
> 코드를 모르더라도, 최종 결과물이 어떤 모습이어야 하는지,
> 어떤 기능이 필요한지, 참고할 레퍼런스가 무엇인지를
> 구체적으로 전달할 수 있다면 — AI가 나머지를 해결합니다.
